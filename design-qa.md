## 可见差异判断

- 投稿弹窗复用现有深色表面、1px 边框、控件圆角、蓝色主按钮和关闭按钮焦点样式，没有引入第二套视觉语言。
- 信息层级从“选择图片 → 名称/分类 → 处理模式 → 提交”自然向下，主操作始终位于右下角。
- iPad 1194px 使用完整三栏工作区；画布、素材面板和属性/图层区没有横向溢出，主要触控目标不小于 44px。
- 弹窗在 834px 高度内完整显示，底部操作未被裁切；背景遮罩、标题、辅助说明和表单间距一致。
- 离线素材由可用的内置/已缓存资源替代远程 URL，不再出现破损缩略图。

## 交互与可访问性

- 主 CTA、素材库/我的提交标签、处理模式、取消/关闭和提交路径均有真实状态逻辑。
- 弹窗支持初始焦点、Tab 焦点限制、Escape 关闭和关闭后焦点恢复；状态更新使用 `aria-live`。
- “我的提交”支持处理中、待审核、已通过、失败、重新提交和跳转素材库状态；终态停止轮询。
- 响应式断点在 CSS 与 JavaScript 中统一为 1180px，并有 1179/1180/1194 边界测试。

## 迭代记录

1. 首轮发现 1194px CSS 已进入桌面布局，但 JavaScript 仍按 1280px 判定平板，导致面板状态冲突。
2. 将 CSS/JavaScript 断点统一为 1180px，并补边界单测。
3. 首轮离线验证发现元数据缓存会生成不可访问的远程缩略图 URL；改为只展示有本地 Blob 的云素材，否则回退内置素材。
4. 将投稿弹窗与现有离线管理弹窗放入同一对照图，复核表面、边框、圆角、按钮、遮罩与间距后通过。

## 最终结论

**通过。** 桌面与 iPad 横屏关键路径没有可见裁切、横向溢出、破图或设计系统偏离；交互和响应式边界均有自动化验证。Windows 原生窗口已启动且系统报告可响应；桌面控制助手截图因系统 `GetCursorPos` 权限被拒，原生画面证据由相同构建入口的浏览器对照图、原生进程状态和后续服务器请求日志交叉补足。
